# WMS 项目开发计划

## 1. 项目定位

项目暂定名称：

- 智能仓储与资产流转管理系统

项目目标：

- 先做一版自己的轻量 WMS，不直接走开源项目二改路线。
- 以本地优先开发为主，先把后端骨架和核心业务闭环做扎实。
- 后续逐步扩展审批流、报表、Docker 部署和微服务演进能力。

当前技术方向：

- 后端：Spring Boot 3 + MyBatis-Plus + MySQL
- 前端：Vue 3 + Vite
- 认证：Sa-Token
- 文档：Swagger / OpenAPI
- 环境：Docker Compose

## 2. 总体策略

当前阶段不直接追求“大而全”，而是按下面的顺序推进：

1. 先把骨架补到够用
2. 再做核心业务闭环
3. 再补企业级增强能力
4. 最后再考虑微服务和分布式演进

核心原则：

- 先做模块化单体，不一开始拆微服务。
- 先跑通真实业务闭环，不先堆复杂架构。
- 优先做能直接帮助后续两周开发效率的基础设施。
- 所有设计尽量为后续拆分预留边界，但不过度抽象。

## 3. 第一阶段：骨架 1.0

目标：把当前项目从“半成品骨架”补成“可持续开发的后端底座”。

### 3.1 必做项

- 统一目录结构
  - `controller`
  - `service`
  - `mapper`
  - `domain`
  - `config`
  - `common`
- 统一接口返回结构
- 统一全局异常处理
- 完成本地配置约定
  - 公共配置放 `application.yml`
  - 本地敏感配置放 `application-local.yml`
  - 模板放 `application-local.example.yml`
- 跑通本地 MySQL
- 保证 Swagger 可访问
- 明确用户、角色、菜单基础模型
- 增加数据库初始化方案
  - 当前可先保留 SQL 脚本
  - 第二步接入 Flyway
- 建立最小可用认证能力
  - 用户名密码登录
  - 登录态校验
- 完善日志与基础审计字段

### 3.2 交付标准

- 项目能稳定本地启动
- 本地数据库能一键拉起
- Swagger 可查看接口
- 至少具备一条完整业务链路
- 新增功能时不需要反复返工底层结构

## 4. 第二阶段：WMS 1.0 核心业务

目标：做出一个可演示、可写进简历、可作为毕设主体的 WMS 核心版本。

### 4.1 优先模块

第一批模块：

- 登录与权限
- 仓库管理
- 库位管理
- 物料管理
- 入库管理
- 出库管理
- 库存查询

第二批模块：

- 库存调整
- 库存盘点
- 调拨管理
- 操作日志
- 基础报表

### 4.2 建议业务闭环

建议优先做这个最小闭环：

1. 新建仓库
2. 新建库位
3. 新建物料
4. 创建入库单
5. 入库后更新库存
6. 创建出库单
7. 出库后扣减库存
8. 在库存列表里查询结果

只要这条链路跑通，项目就已经从“后台骨架”进化成“真实业务系统”了。

## 5. 第三阶段：企业化增强

目标：把项目从“学生作品”提升成“企业中后台风格项目”。

建议增强项：

- RBAC 权限模型完善
- 菜单与按钮权限
- 数据权限
- 审批流
- 导入导出
- 操作留痕
- 登录日志
- 业务日志
- 统一字典管理
- 文件上传
- OSS 对接
- Redis 缓存
- 定时任务
- 报表统计

这一阶段重点是提升项目的“企业味”和“可讲述性”。

## 6. 第四阶段：部署与演进

目标：让项目具备上线和继续扩展的能力。

### 6.1 部署方向

- 本地使用 Docker Compose
- 服务器部署使用 Docker Compose
- GitHub Actions 做 CI/CD
- `main` 分支自动部署
- 其他分支只做测试和构建

### 6.2 演进方向

当下面条件逐步出现时，再考虑拆服务：

- 某个模块边界已经非常清晰
- 某个模块发布频率明显更高
- 某个模块需要独立扩容
- 团队协作开始按模块分工

优先可拆的候选模块：

- 用户认证中心
- 文件服务
- 消息通知服务
- 审批服务

## 7. 当前建议的开发顺序

下面是建议的实际落地顺序：

1. 补骨架 1.0
2. 完成登录接口
3. 完成用户/角色/菜单基础表
4. 完成仓库管理
5. 完成库位管理
6. 完成物料管理
7. 完成入库单
8. 完成出库单
9. 完成库存查询
10. 完成调拨和盘点
11. 补日志、导出、报表
12. 做 Docker 化与服务器部署

## 8. 当前阶段不建议投入过多的内容

以下内容先不要做太重：

- 微服务拆分
- 分布式事务
- 注册中心
- 配置中心
- 链路追踪全家桶
- 复杂工作流引擎
- 大量通用基类和过度封装

原因：

- 会明显拖慢开发速度
- 会增加理解和维护成本
- 对当前 1.0 业务闭环帮助不大

## 9. 近期两周执行清单

### Week 1

- 收口骨架
- 补登录模块
- 明确用户、角色、菜单表
- 跑通 Swagger
- 跑通本地 Docker MySQL

### Week 2

- 完成仓库管理
- 完成库位管理
- 完成物料管理
- 开始入库单设计

## 10. 阶段性完成标准

### 骨架完成

- 能启动
- 能连接数据库
- 能登录
- 能看 Swagger
- 具备基础用户权限模型

### WMS 1.0 完成

- 能完成入库、出库、库存查询闭环
- 有基础权限控制
- 有较清晰的数据结构
- 可以演示主要流程
- 可以作为毕设主体说明

### 企业增强版完成

- 有审批和日志
- 有导出和报表
- 能部署到服务器
- 能作为简历项目重点介绍

## 11. 备注

现阶段最重要的判断标准只有一个：

如果某项工作不能明显帮助当前版本更快做出核心业务闭环，就先不做。

先把“自己的 WMS 1.0”做出来，再谈更重的架构升级。
